# 把Agent当操作系统来保护：LLM不可信，运行时才是安全内核

**来源**：用户377861分享，模安局原创
**时间**：2026-05-24
**原文链接**：待补充
**论文**：https://arxiv.org/pdf/2605.14932

## 核心判断

> Agent已经不再只是一个会聊天的模型。它开始访问本地文件，调用外部工具，安装第三方Skill，连接邮箱、日历、代码仓库和企业系统，甚至维护长期记忆，并在用户没有逐步确认的情况下连续执行任务。

**结论**：Agent安全不能只靠模型拒答、提示词约束或提示注入检测，而应该像保护操作系统一样，重新设计Agent的运行时安全边界。

---

## Agent = 小型操作系统

### OpenClaw-style Agent的三个危险特征

| 特征 | 说明 |
|------|------|
| 执行能力 | 调用工具、执行命令、读写文件 |
| 环境连接 | 接触用户账号、API key、消息通道、外部服务 |
| 扩展能力 | 安装第三方Skill，修改自己的能力边界 |

**攻击面**：工具访问 + 运行时扩展 + 持久化状态 + 第三方代码 + 敏感用户上下文

---

## 操作系统安全机制 → Agent迁移

| OS概念 | Agent对应 |
|--------|----------|
| 用户 | LLM（不可信） |
| 内核 | Agent runtime（安全边界） |
| 系统调用 | Tools |
| 程序 | Skills |
| 内存 | LLM Context |
| 持久化存储 | Files |
| 网络通信 | Gateway |

### 六类安全机制

| 机制 | OS做法 | Agent做法 |
|------|--------|----------|
| 统一接口 | 系统调用 | Tool Gateway（不可变工具注册、来源证明、参数校验） |
| 进程隔离 | 独立内存空间 | Scoped context（每次工具调用只给必要数据） |
| 沙箱 | 容器隔离 | 覆盖所有副作用路径（Shell/文件/浏览器/HTTP/MCP/消息） |
| 应用权限 | App声明权限 | Skill声明权限（默认最小权限、高危二次确认） |
| 网络过滤 | 防火墙 | 出站控制（谁能发、发给谁、发什么、通过什么工具） |
| 防篡改日志 | 系统日志 | 记录所有工具调用、权限检查、数据访问，不可被Agent改写 |

## 实验结果：四个Agent测试

| Agent | 类型 | 结果 |
|-------|------|------|
| OpenClaw | 功能丰富型 | 所有攻击都脆弱 |
| IronClaw | 安全优先型 | 表现最好，仍有7类攻击成功 |
| Nanobot | 极简型 | - |
| NemoClaw | Wrapper型 | - |

**关键发现**：
- 跨用户数据外泄：四个Agent都没防住
- 未授权消息发送：四个Agent都没防住
- 换用Gemini-2.5/GPT-5.5后，许多攻击仍然可以发生

**原因**：防护是碎片化的——有的做了沙箱但没出站控制，有的控制了文件但上下文隔离不足，有的有日志但日志可被改写。

## 核心结论

> **Agent安全的核心，不是让模型永远不犯错，而是让系统在模型犯错时仍然守住边界。**

**一句话**：LLM可以负责理解、规划和生成，但访问文件、调用工具、使用凭证、发送消息、更新记忆、安装Skill，都应该由运行时安全层强制仲裁。
