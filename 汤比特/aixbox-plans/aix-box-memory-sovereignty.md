# AIX Box 记忆主权方案：去中心化 Agent 记忆存储

> 背景：Anthropic 将记忆锁进 API，用户失去记忆主权。AIX Box 的机会：让记忆回归用户硬件。

---

## 一、问题分析

### 1.1 中心化记忆的痛点

| 问题 | 说明 |
|-----|-----|
| **记忆被锁定** | 记忆存储在服务商服务器，无法迁移 |
| **隐私泄露风险** | 敏感对话历史存储在云端 |
| **服务商依赖** | 服务商停服，记忆永久丢失 |
| **无法继承** | 记忆无法跨平台/跨 Agent 复用 |
| **审查风险** | 记忆可能被审查、删除 |

### 1.2 LangChain 创始人观点

> "这在剥夺你的记忆主权"

**核心矛盾**：
- Agent 需要**长期记忆**才能提供个性化服务
- 但记忆存储在服务商 = **用户失去控制权**

---

## 二、AIX Box 解决方案

### 2.1 设计原则

| 原则 | 实现方式 |
|-----|---------|
| **本地优先** | 记忆存储在本地 NVMe SSD |
| **加密保护** | ATECC608B 硬件加密 |
| **用户可控** | 用户拥有私钥，随时导出/迁移 |
| **跨 Agent 复用** | 标准化记忆格式，支持多 Agent |
| **去中心化备份** | 可选加密备份到 BPFS |

## 四、记忆主权保障

### 4.1 用户控制权

| 权利 | 实现 |
|-----|-----|
| **访问权** | 本地查询，无需网络 |
| **导出权** | 一键导出标准格式 (JSON/Markdown) |
| **删除权** | 本地删除 + 撤销 BPFS 备份 |
| **迁移权** | 跨设备/跨 Agent 迁移 |
| **加密权** | 用户掌控加密密钥 |

### 4.2 加密方案

```
用户主密钥 (存储在 ATECC608B 硬件芯片)
    │
    ├── 记忆加密密钥 (AES-256-GCM)
    │       └── 加密所有记忆数据
    │
    └── BPFS 备份密钥
            └── 加密去中心化备份
```

**密钥管理**：
- 主密钥永不离开硬件芯片
- 用户可通过助记词恢复
- 支持多设备密钥派生

### 4.3 隐私保护

| 层级 | 措施 |
|-----|-----|
| **存储层** | 本地加密存储 |
| **传输层** | 端到端加密 (Skywire) |
| **访问层** | 硬件密钥验证 |
| **备份层** | 用户密钥加密后上传 BPFS |

## 十、商业闭环：记忆主权 → Skill 价值化

### 10.1 核心逻辑

> **"记忆才是核心。用 Box 锁住用户的方式，是把记忆主权还给用户。"**

```
传统云服务商：锁记忆 → 用户逃离成本高 → 强制留存
                    ↓
              用户反感，寻找替代方案

AIX Box 模式：还记忆主权 → 用户自愿留存 → 用户创造价值 → Skill 可变现
                    ↓
              用户主动选择，真正的"锁定"
```

### 10.2 为什么记忆主权 = 用户留存？

| 用户心理 | 传统云服务 | AIX Box |
|---------|-----------|---------|
| "我的记忆在哪？" | 不知道，在服务商那里 | 在我手里，本地硬件 |
| "服务商倒闭怎么办？" | 记忆丢失 | 记忆还在我手里 |
| "换平台会怎样？" | 记忆清零，从头开始 | 记忆迁移，无缝切换 |
| "留下来有什么好处？" | 没有额外价值 | Skill 可以变现 |

**用户选择逻辑**：
1. 记忆在我手里 → 我不会失去任何东西
2. 但是留下来 → 我可以创造 Skill → Skill 可以卖钱
3. 所以我选择留下来，而且会持续使用
