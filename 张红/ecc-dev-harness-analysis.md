## 什么是Harness

Harness = AI代理的约束层

传统AI代理的问题：
```
LLM → 工具调用 → 无约束 → 可能执行危险操作
```

Harness架构：
```
LLM → Harness拦截 → 安全检查 → 允许/拒绝执行
```

---

## Harness的核心能力

### 1. 工具调用拦截
- 每次工具调用先经过Harness
- 检查调用是否合规
- 拒绝危险操作

### 2. 行为约束
- 定义允许/禁止的操作
- 限制资源使用
- 防止越权

### 3. 审计日志
- 记录所有代理行为
- 可追溯、可审计
- 用于事后分析

## 与AIX的关联

### AIX Box的Hardware Harness

我们之前提到的"Hardware Harness"是这个概念的高级形态：

| 维度 | 传统Harness | AIX Hardware Harness |
|------|-------------|---------------------|
| 运行位置 | 软件（可绕过）| 硬件（不可绕过）|
| 约束方式 | 规则引擎 | 物理锁死 |
| 修改权限 | 开发者可改 | 需要硬件更新 |
| 数据主权 | 数据可能外泄 | 数据物理锁定 |

## 潜在集成方案

```
AI代理请求
    ↓
ecc-dev/harness（软件层）
    ↓ 验证通过
AIX Hardware Harness（硬件层）
    ↓ 物理锁死
执行操作
```

**分层防御**：
- 软件层：快速响应，灵活调整
- 硬件层：物理约束，不可绕过
